Introduction

The purpose of this document is to provide the detailed design decisions made by the firm's system designer(s). This document's primary intended audience(s) is the developer's team, who will execute this SDS in the project's implementation stage. This SDS primarily provides the class and database design diagrams and how the components communicate and coexist within the whole software. These designs are subject to change in each iteration if needed, depending on the development framework the project manager is using, such as agile, waterfall, etc.

Description of Design Components

There are multiple design components of the system design: server-side "backend," client-side "frontend" (web app and mobile app), and third-party services.

Backend: The backend is the core brain of the software where all the business logic resides. Only the server-side "backend" can interact with the database using an Object Relational Mapper. All non-third-party software services will be exposed using RESTFul APIs. Web and mobile app frontends will use those APIs to interact with the software's backend. For the backend solution, NodeJS with ExpressJS has been selected.

Frontend: The end-user interfaces, or the client-side "frontend," are divided into web application frontend and android applications frontend. Although there are restrictions on the android app's use-cases within the software, both frontend will use the exposed APIs from the backend to give the end-users a modern frontend experience. Using the same backend server will lead to data sync among interfaces. For the frontend solution, ReactJS is selected for the web application. For the android application, official Android Native with Kotlin will be used.

Database: Because the data in the software is relational, this system designer(s) has decided to use a Relational Database. This software will use an ORM(Object Relational Mapper) for best practices. For the ORM solution, Sequelize has been chosen. Because Sequelize allows different dialects(Postgres, MySQL, MariaDB, SQLite, etc.) in different environments, the design doesn't depend on any single database solution.

Third-Party Services: The Google Cloud's APIs have been chosen for the OAuth solution within the software. Furthermore, for a text-to-speech solution, AssemblyAI has been selected. The backend will connect to these Third-Party Services through API calls.
