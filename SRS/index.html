Introduction

Purpose

This document outlines the initial revision of the software requirements for the exam cohort system for conducting online exams. This document serves its intended users a full unclouded version of the client's requirement about the software, written down by the business analyst team being as specific as possible in every possible way. This document covers the entire exam-cohort system's requirements, not just a subsystem or part of the system.

Document Conventions

Intended Audience and Reading Suggestions

The document's intended audience is Business Clients, System Design Engineers, Quality Assurance Engineers, Project Managers, and future maintenance-period team(s). The business analyst can provide the SRS to the client to make sure the requirements are, in fact, correct; if not, then they can bridge the gap, hence saving time, cost, and resources. After fully understanding the project's requirements, the system designer(s) can plan their SDS. The project manager can grasp all the requirements, plan the project, and choose appropriate development strategies such as agile, waterfall, or others. Hence, the firm can ship out high-priority use cases as a priority, eventually saving resources, time, and costs. The quality assurance team will engineer a test plan according to the requirements of the SRS document to test that the implemented software meets the client's requirements. As a fail-safe strategy, this document is intended for maintenance-period teams who want to understand the software's purpose, scope, parameters, and goals when the initial project team is away. However, it is recommended to restrict the access of this document to software developers, as doing so will help developers rely only on SDS. As a result, system design errors or inefficiency will be caught in the testing phase by the Quality Assurance team. Hence, revisions in the system design can be made possible in the next iteration. As a reading suggestion, following the order of indexes provided in the "Table of Contents" of the SRS document is recommended.

Product Scope

The "Exam-Cohort" system provides the opportunity to take online exams. The software system's every user has two roles: evaluator and candidate, in different circumstances. Users act as evaluators when creating assessments; however, they act as candidates when appearing for an assessment. One of the primary objectives of the software is to provide the users with a platform that replaces physical exams. However, the key business differentiator of this software is the micro-viva feature. The micro-viva questions are presented to the candidates as a voice recording of the evaluator. In counterpart, the candidates will answer the micro-viva by recording their voice, a feature built within the system. The software will maintain security issues so that the exam system remains as secure as possible while preventing accidental "answers" leaks. The software complements any learning management system, educational institute, or modern-day EdTech business; as a result, it can be incorporated with those businesses or institutes.

Overall Description

Product Perspective

This product is a part of the software engineering course assessment at North South University's summer academic semester of 2022. The faculty member proposed the project idea while acting as a "client" with the student group members who performed the tasks as an imaginary software firm. The product is a new self-contained system that is neither a member of a product family nor a replacement of another system.

Product Functions

The product must let users log in to the system using their google account, meaning there will be no separate signup system. Each user, while acting as an evaluator, must be able to create an exam cohort and add other users as a candidate to this cohort. Evaluators can add assessments within the cohort; meanwhile, each assessment is capable of having multiple questions. Users who are added to a cohort as candidates can appear for assessments. The system will automatically mark the assessment and generate a spreadsheet with complete breakdowns of each mark of each assessment for each student. This auto-generated spreadsheet can be accessible by the evaluator. After the assessment is finished, evaluators can re-evaluate a candidate's answers to re-adjust the marks. When a candidate takes the assessment, the system can detect "time limits exceeds" scenarios. As a response, the system can perform actions such as moving to the next question or auto-submitting the assessment if it is the last question as a part of security measures.

User Classes and Characteristics

Every user is capable of logging into the system using a google account, meaning there will be no separate signup system. So, when a user logs into the system, they will be registered into the system if not so.

Each user must be able to create an "Exam-Cohort." The user who creates the exam-cohort" is the "Evaluator" of that cohort. And within each exam-cohort, the evaluator can add "other users of the system" as a candidate. So, the candidates have to exist in the system when the evaluator is assigning candidates.

Within each exam cohort, the evaluator, besides adding candidates to that cohort, has the right to create assessment(s). Furthermore, within each assessment, an evaluator can add MCQs or Micro-viva questions, re-adjust the marks of a student, and provide the necessary information for that assessment (i.e., available date-time and due-date-time). Within each question, the evaluator must provide the information required for that question(i.e., marks and time limit).

MCQs are the standard multiple-choice-questions, capable of having multiple correct answers, so the candidate must select all the answers to gain marks in that question. As micro-viva questions and answers are entirely in the form of voice recording, evaluators and candidates will be entitled to record or listen to the recorded voice within the user interface provided by the software. When adding micro-via questions or adding the correct answer to a micro-viva, the evaluator will record their voice. When sitting for an assessment, the candidates can hear the playback of the micro-viva questions and provide their answers in a recorded voice. When the evaluator wants to re-adjust a candidate's assessment marks, he can listen to micro-viva answers and view MCQ answers.

The system will automatically mark the assessment and generate a spreadsheet with complete breakdowns of each mark of each assessment for each student. The evaluator will have access to that spreadsheet.

Operating Environment

The software will operate on web app and android devices. The operating environment should have access to an microphone for voice recording. It is recommended to not use any old Android OS version and use a relatively modern web browser.

Design and Implementation Constraints

The client requested that the software should use relational database systems, as the data are relational. However, if the software ever uses any non-relational database system, it should follow industry standards and best practices: DRY, KISS, and SOLID. Also, the client requested that the software's micro-vivas voice recordings need to be evaluated using external API solution(s) with correct answers. The software should have a language boundary of the English language.

For security considerations, the candidates have the right to only appear once for a particular assessment of an exam-cohort assigned by the evaluator of the cohort. Candidates cannot access an exam cohort that an evaluator did not assign them. The same goes for assessments not part of any exam cohort from the candidate's assigned cohort lists. Furthermore, the questions and answer options a candidate sees when appearing for an assessment will be randomized and presented in such a manner. There should be no correct answer leaks from any interfaces interacting with the candidates. A candidate can never have the advantage of going past the question's time limit. If the time exceeds for a question, the system automatically detects that. When the system detects such cases, the next question will be presented, withdrawing any access to the previous question from the user interface. When time exceeds, and if there is/are no next question, the system will submit the assessment automatically. A question's time limit constraints also extend to the micro-viva type questions, and the candidate must record and submit their recorded answers within the time limit.

User Documentation

The software comes with a user manual and a short video for non-technical individuals that explains "how to install and use the software."

Assumptions and Dependencies

The software assumes that the user's devices will have access to the internet that can connect to the internet, have a google account, is capable of using a web app and android app, and have access to necessary devices to use those interfaces. It is also assumes that the user have access to a microhphone that can record voices. This software relies on APIs such as Google OAuth API and external ML speech-to-text APIs, So the software's one of the core differentiators relies on the external ML model's performance, and it is assumed that the ML model's performances are reasonable. The project could be affected if these assumptions/dependencies are incorrect, are not shared, or changed.

External Interface Requirements

User Interfaces

The system will first take the user to the login screen to log in or register with google. After successfully logging in, the user sees the list of Exam-Cohort they are enrolled in or have created. There will be a button to create an exam cohort; upon pressing the button, the system will provide the user with a form to fill out the necessary information and add the candidates. Users can create as many assessments as they wish in any exam cohort. There can be two types of questions, MCQs and Micro-viva's. To set the MCQ question, the evaluator(s) will provide the question and answer in the text; to put the micro-viva, they will have to provide both questions and answer recording. The candidates must respond within the given time. Upon finishing an exam, the system will evaluate the answers automatically, and the evaluator can adjust the mark and download the spreadsheet prepared by the system. In the android application, users can attend both micro-viva and MCQ assessments.

Hardware Interfaces

The primary hardware requirement for the system is a computer or a mobile phone. The system will require microphone permission to assign and give micro viva's and an internet connection to call API.

Software Interfaces

This system will use ORM to map relational cardinalities. It will use a Speech-to-text machine learning model API to detect questions and answers for the micro viva. This system will also use Google OAuth 2.0 to log in or register to the app with google.

Communications Interfaces

The system will use Restful API to connect the frontend with the backend. This system uses OAuth 2.0 by Google for verification and token authentication to communicate between the frontend and backend through API.

System Features

Login

Use Case Login using Google Account
Actors User
Purpose Authenticate the user to the exam-cohort system using a google account.
Overview The individual decides to log into the software using the user interface. To do that, they click the "Sign in with Google," and a Google login page is presented to the user. When the user successfully logs into their Google account, they are redirected to the software's user interface, and a message is relayed. The system will notify the user if the logging event is a success or failure. Upon successful authentication, if the user was new to the system (i.e., they never logged into it), the system will create a brand new account in the database. However, suppose the user has an existing account. In that case, the software will load that particular account's details (such as saved exam-cohorts, assessments, etc.) into the software for the user.
Type Primary
Cross References Not Applicable.
Typical Course of Events
Actor Action System Response
1. This use case begins when the user chooses to log into the system.
2. The user clicks/touches the "Sign in with Google" button from the user interface. 3. The system responds by throwing a google login form as a pop-up window in the web app or a dialogue box in the android app.
4. The user fills/selects their necessary details and logs into their correct google account using the google OAuth login form. 5. The system awaits for the OAuth form and detects if the OAuth was successful.
6. After successful authentication, if it is a returning user, the system will use previous user profile data(such as exam-cohorts, assessments, etc.) However, the system will generate a new profile if it is a new user. Then the system will generate the "token authentication"'s' token and provide this token to the user interface.
7. After getting the necessary tokens, the system will redirect the user to their dashboard, where they can see all their data (such as exam-cohorts and others) and relay the message "Successfully logged in."
Alternative Courses
  • Section 3: If google is not responding or is having an outage, then the system will show a message "Please try again later!".
  • Section 4: If the Google OAuth login form is closed without completing the authentication, the message "Login failed!" is relayed to the user interface.

Appear in Assessments

Use Case Appear in Assessments
Actors User
Purpose Users who are candidates of cohorts appears in the assessments of that cohort.
Overview When a user is added as a candidate in a cohort, they can appear/sit for the assessment. The system provides an interface for an online exam with strict time limits. The system also provides all the necessary user interfaces to the candidates for the exam, such as voice recording and a listening mechanism in the user interface for the micro-vivas. If the time limit exceeds, the user can no longer access that question and is forced to move to the next question or end the assessment if no other question remains. The system presents all the answers and options in randomized order.
Type Primary
Cross References Functions: Section 4.1, 4.3, 4.4, 4.9
Typical Course of Events
Actor Action System Response
1. This use case begins when the user wants to appear for an assessment of a cohort.
2. The user clicks the "Take Assessment Now" Button in the assessments panel. 3. The system takes the user to the exam-taking user interface that shows a random question from the assessment with randomized options if that is an MCQ question. The system provides a user interface with a playback player and a voice recorder if it is a micro-viva-type question. The system also shows the current question's time limit counter.
4. The user answers the question; if it is MCQ, they select option(s). If it is a micro-viva, they listen to the question's recording and answer with a recording via the interface. The user clicks the "Next" button when the answering is completed. 5. The system saves the candidate's answers in the database, presents the next random unanswered question from the assessment's questions list, and repeats from Section 4 until no unanswered questions are left.
6. The system prompts the user with a message: "Congratulations, you have completed the assessment!" and redirect the candidate to their dashboard.
Alternative Courses
  • Section 1: If the user is not logged in, the system redirects them to the login page and relays a message "You must be logged in!" in the user interface.
  • Section 2: If the user is not a cohort candidate, then the system will redirect the user to the dashboard and relay a message "Access Denied! You are not the candidate of that cohort!" in the user interface.
  • Section 4: If the time limit is exceeded for that question, the system detects that. The system then forces the user to move to the next question, and access to the previous question is withdrawn from the user interface.
  • Section 4: If the user does not answer a question and tries to go to the next question, the system prompts the user of possible danger: "Are you sure you want to skip this question?" and receive a yes or no response from the user and act so.
  • Section 5: If the current system time is not between the range of available date-time, and due date-time, the system will not save the answers in the database and prompt with "Time Exceeded for the Assessment!" and redirect the user to the dashboard.

Actors User
Purpose Users who are evaluators can evaluate the assessments within the exam cohort.
Overview When a candidate has already appeared in the assessment, the system marks them automatically. But when the evaluator thinks there was a problem with the provided answer, the evaluator can adjust the mark for all students. The evaluator can also see/hear the script of any candidate within the exam cohort.
Type Primary
Cross References Functions: Section 4.1, 4.2, 4.3, 4.4, 4.5, 4.8
Typical Course of Events
Actor Action System Response
1. This use case begins when the user wants to adjust the mark or see the script of a candidate.
2. The user clicks the "Check Response" Button in the assessments panel. 3. The system presents the list of students and their marks with a button named "Adjust Mark"
4. If the user(evaluator) wants to check the answer script of any candidate, the user clicks/taps on that candidate's name. 5. The system shows the answer script of that candidate. The answer script will not be randomized, and there will be playback buttons for the micro viva question and answer.

  • Section 2: If the assessment didn't start, the evaluator would be prompted with a message, "Assessment not started yet!".
  • Section 2: If the evaluator added no candidates in the exam cohort, the system would prompt the evaluator with a message, "No candidates found."
  • Section 4: If the given Candidate didn't appear in the assessment, the evaluator would be prompted with a message, "Candidate didn't appear in assessment."
